Introduction

The Moorfields Eye Hospital Project Board asked the OpenEyes team to evaluate the impact of a temporary pause in new development work. The purpose of this scheduled break is to address some of the challenges facing the product within its current architecture.

The primary issues are listed below, please click on each issue for more detail:

  • Slow development speed The development team cannot scale up as modules are interdependent
  • Inefficient development Change in clinical workflows has widespread implications, making development inefficient
  • Reduced configurability The solution is not easily configured to meet customers’ requirements, including Moorfields’ satellites
  • Lack of integration with other systems Having the information that is needed in one place, such as imaging, is not currently possible
  • Difficult to deploy and to support Costly to configure, deploy and support in different clinical environments due to lack of configurability
  • User experience is hindered by the architecture User journeys are not representative of behaviour as role based, patient centric workflows are not supported

The OpenEyes team recommended a 7 week Proof of Concept exercise to highlight a range of technical concepts that will provide the architectural solution OpenEyes needs to overcome these challenges.

Architecture

This is a high level view of the target architecture we have demonstrated in the Proof of Concept.

  • Application Frontend Built in Angular, which is a JavaScript framework maintained by Google. It has a very active user base for contributions and is set up to facilitate separation of concerns, so enables modularity. Allows multiple clients on multiple devices (tablet, PC, mobile)
  • JVM Based Service Layer A service orientated architecture allows developers to work in parallel, makes it easier to accommodate change and integrate with other systems and devices.
  • Document data store Efficient information storage putting the patient at the centre of care, providing a view of the patient rather than the condition

Benefits target architecture enables

Please click on each benefit for more detail:

  • Scalability of the development team Enabled by a service layer architecture, with different teams working on different areas in parallel.
  • Development speed Increased by de-coupling modules of functionality and developing in Java-based language such as Scala; furthermore by using Angular as a front-end.
  • Workflow, Clinical scalability, Configurability and Deployment in multiple organisations Workflow defined by user role, location and speciality. Supports MEH transformation programme, allowing clinic workflows to be configured using our Workflow API to reflect practises of/within different organisations.
  • Modularity Workflows are built comprising of modular elements which can easily be added, ordered and removed.
  • Integration Technology choices give access to integration libraries which enable the proof of concept to easily connect to other systems and devices.
  • Platform scalability A service oriented architecture with an HTTP API at the centre provides a more scalable platform which can be built upon without breaking other components

Workflow

The benefits of this architecture are showcased through an example patient journey that is not well defined in OpenEyes today: a patient with Age Related Macular Degeneration.

This journey is representative of 60% of patients at the Medical Retina Injection Clinic. Over the course of their care during a visit, the patient meets several different clinicians in the following steps:

  • Registration
  • Vision Assessment
  • OCT Scan
  • Assessment for AMD
  • Injection

Please note:

This patient journey is solely intended to showcase the platform for the re-architecture and is not ready to be launched for use by Clinicians. It is NOT intended to show a redesign or rebrand, this will be one of the top priorities for the next phase of the project.
